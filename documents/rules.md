# Rules Domain Model

## Purpose

This document defines what a rule is in DIAL-IN and how rules relate to tasks, categories, and future scheduling work.

The goal is to keep the boundary between recurring automation and ad-hoc task management clear.

---

## Core Model

### Rule

A rule is a recurring task blueprint plus a schedule.

It answers:
- What task should be generated?
- On what pattern should it be generated?
- What category and metadata should generated tasks inherit?

Rules are not task attachments. They are sources of future task instances.

### Task

A task is a concrete work item.

There are two task origins:
- Manual task: created directly by the user and not derived from a rule
- Generated task: created automatically by a rule for a specific occurrence

### Rule-to-Task Relationship

`task.rule_id` should mean one thing only:

This task was generated by this rule.

It should not mean:
- this task is merely associated with a rule
- this task was manually created and later converted into a recurring template
- this task should start recurring from now on

---

## Product Rules

1. Users may create ad-hoc tasks from the Tasks workflow.
2. Users may create recurring rules from a dedicated Rules workflow.
3. Manually created tasks must not be assigned a `rule_id`.
4. Only automatically generated tasks may carry a `rule_id`.
5. Existing tasks should not be retroactively linked to rules.
6. A rule is not created from within an existing task instance.
7. Generated tasks may display their source rule, but that linkage is provenance, not a general-purpose editable association.

---

## Why This Model

This separation keeps the system conceptually clean.

### Benefits

- Clear provenance: a linked rule means the task came from automation
- Cleaner UX: task editing and rule editing have different purposes
- Cleaner data model: no ambiguity around whether a task is a template or an instance
- Easier future scheduling logic: rule processing only creates new tasks rather than mutating existing ones into recurring objects
- Fewer edge cases: no attaching a mismatched or unrelated rule to an existing task

### Avoided Confusion

Without this boundary, the system blurs three different ideas:
- a template for future work
- a scheduling rule
- an actual task instance

Those should remain distinct.

---

## Category Behavior

Rules and tasks may both have category data, but their roles differ.

- A rule category defines the default category for tasks it generates.
- A generated task may inherit the rule category at creation time.
- A manual task category is chosen directly by the user.

Under this model, category conflict between a task and a rule should not come from the normal workflow because manual tasks are not attachable to rules after creation.

Future implementation should decide whether generated tasks:
- keep a copied category value that can later diverge from the rule, or
- remain constrained to rule-derived values

For now, the important invariant is that manual task to rule linking is not allowed.

---

## Editing Semantics

### Editing a Manual Task

Edits affect only that task.

### Editing a Generated Task

Edits affect only that generated occurrence unless a future feature explicitly introduces series-level editing.

### Editing a Rule

Edits affect future generated tasks, not historical or already-created task instances.

This is the same blueprint-versus-instance boundary used in calendar and recurrence systems.

---

## UI Guidance

### Tasks Page

The Tasks experience should remain focused on task instances.

It should support:
- creating manual tasks
- editing manual tasks
- viewing generated tasks
- optionally showing that a task was generated from a rule

It should not support:
- converting a manual task into a rule
- attaching an existing rule to a manual task
- creating a new rule from inside an existing task card

### Rules Page

The Rules experience should be the place for:
- creating rules
- editing rules
- enabling or disabling rules
- defining recurrence patterns
- previewing future behavior once calendar support exists

## Rate Pattern Scope

Rules use the recurrence encoding documented in [rate_encoding.md](./rate_encoding.md).

Supported documented recurrence types are:
- Daily (`d`)
- Weekly (`w`)
- Monthly by date (`m`)
- Monthly by weekday occurrence (`mw`)
- Yearly (`y`)

Optional supported components are:
- Month filter (`M`)
- Time component (`T`)
- Multiple rule segments separated by semicolons

The rule builder UI should target this encoding model.

---

## Implementation Constraints

### Current Meaning of `rule_id`

The backend already allows `Task.rule_id`, but product behavior should constrain its use.

Desired rule:
- `rule_id` is written only by task generation logic, not by standard manual task create or edit flows

### Validation Direction

When rules engine work begins, backend validation should enforce at least:
- manual task creation does not accept arbitrary `rule_id`
- manual task editing cannot add or swap `rule_id`
- generated tasks can persist the originating `rule_id`

### Creation Flow Direction

Preferred long-term flows:
- Create Task -> manual task only
- Create Rule -> recurring blueprint only
- Rule execution -> creates generated tasks

---

## Deferred Work

These areas remain open for future decisions:
- whether generated tasks can detach from their source rule
- whether generated task fields are partially locked or fully editable
- whether rule edits can optionally propagate to future uncompleted generated tasks
- how future calendar previews should visualize upcoming rule-generated tasks
- whether rules can generate events in addition to tasks

---

## Summary

Rules are templates with schedules.

Tasks are instances.

Manual tasks stay manual.

Generated tasks carry rule provenance.

The UI and backend should enforce that distinction rather than allowing rules to be attached to tasks after the fact.
